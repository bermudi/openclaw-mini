# session-compaction Specification

## Purpose
Automatic and manual summarization of session message history to prevent context window overflow while preserving conversational continuity.

## Requirements

### Requirement: Automatic session compaction at message threshold
The system SHALL automatically trigger session compaction when the number of `SessionMessage` rows for a session exceeds a configurable threshold (default: 40). Compaction SHALL summarize the oldest messages (all but the most recent N, default: 10) into a single `system`-role summary message. The original messages that were summarized SHALL be deleted after the summary is inserted.

#### Scenario: Session hits 40 messages and compaction triggers
- **GIVEN** a session has 40 `SessionMessage` rows
- **WHEN** a new message is appended via `appendToContext`
- **THEN** the system SHALL summarize the oldest 31 messages into a single summary message, delete those 31 rows, insert the summary as a `system`-role `SessionMessage`, and retain the 10 most recent messages plus the new message intact

#### Scenario: Session below threshold does not compact
- **GIVEN** a session has 25 `SessionMessage` rows
- **WHEN** a new message is appended
- **THEN** no compaction SHALL occur and all 26 messages SHALL remain as individual rows

#### Scenario: Compaction preserves the existing summary
- **GIVEN** a session was previously compacted and contains a summary message from the prior compaction
- **WHEN** compaction triggers again
- **THEN** the prior summary message SHALL be included in the content sent to the model for re-summarization, ensuring cumulative context is not lost

### Requirement: Manual compaction endpoint
The system SHALL expose a `POST /api/sessions/:id/compact` endpoint that triggers compaction on demand, regardless of message count. The endpoint SHALL return the number of messages summarized and the resulting message count.

#### Scenario: Manual compaction via API
- **WHEN** a POST request is made to `/api/sessions/:id/compact` for a session with 35 messages
- **THEN** the system SHALL compact the oldest 25 messages into a summary, retain the 10 most recent, and return `{ summarized: 25, remaining: 11 }`

#### Scenario: Manual compaction on a small session
- **WHEN** a POST request is made to `/api/sessions/:id/compact` for a session with 5 messages
- **THEN** the system SHALL return a response indicating no compaction was needed (fewer messages than the retention count)

### Requirement: Summary message preservation
Summary messages generated by compaction SHALL have a `role` of `system` and SHALL include a prefix marker (e.g., `[Session Summary]`) so the system can identify them as compaction artifacts. Summary messages SHALL NOT be deleted by subsequent compactions without being included in the new summary input.

#### Scenario: Summary message is identifiable
- **WHEN** compaction produces a summary message
- **THEN** the message content SHALL begin with `[Session Summary]` and the `role` SHALL be `system`

### Requirement: Pre-compaction memory flush
Before deleting summarized messages during compaction, the system SHALL flush the raw content of those messages to the agent's memory history via `memoryService.appendHistory`. This ensures the full conversational detail is preserved in the memory archive even after compaction removes it from the session.

#### Scenario: Compaction flushes to memory before deleting
- **GIVEN** compaction is about to summarize and delete 30 messages
- **WHEN** the compaction executes
- **THEN** the system SHALL call `appendHistory` with the raw content of those 30 messages BEFORE deleting them from the `SessionMessage` table

### Requirement: Compaction during active conversation
Compaction SHALL NOT block incoming messages. If a new message arrives while compaction is in progress, it SHALL be appended normally. The compaction process SHALL only delete and summarize messages that existed at the time compaction started (snapshot isolation).

#### Scenario: Message arrives during compaction
- **GIVEN** compaction has started and is summarizing messages 1–30
- **WHEN** a new message (message 42) is appended during the summarization
- **THEN** message 42 SHALL be inserted as a normal `SessionMessage` row and SHALL NOT be included in the current compaction's summary or deletion set
